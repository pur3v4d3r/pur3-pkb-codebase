### Query 2: Obsidian Tutorials by Maturity Level

```
TABLE maturity, status, file.mtime as "Updated"
FROM #type/tutorial AND #obsidian
SORT maturity ASC, file.name ASC
```

**Use-case:** Find beginner-friendly tutorials when helping others, or advanced tutorials for your own learning.

### Query 3: System Maintenance Tasks

```
TASK
FROM #pkb/maintenance OR #knowledge-workflow/review
WHERE !completed
SORT priority DESC
```

**Use-case:** Track pending maintenance and review tasks for your PKB.

### Query 4: Recent PKM Experiments and Practice

```
TABLE date, tags as "Focus", insights
FROM #type/practice-log
WHERE contains(file.tags, "pkm") OR contains(file.tags, "knowledge-workflow")
SORT date DESC
LIMIT 15
```

**Use-case:** Review your recent experiments with PKM techniques and systems.

### Query 5: Evergreen vs. Developing Notes

```
let pages = dv.pages('#pkm OR #pkb OR #obsidian OR #knowledge-graph')
let statusCount = {}
for (let page of pages) {
    let status = page.status || "untagged"
    statusCount[status] = (statusCount[status] || 0) + 1
}
dv.table(["Status", "Count"], 
    Object.entries(statusCount).sort((a,b) => b[1] - a[1])
)
```

**Use-case:** Monitor the maturity distribution of your PKM documentation.

```

## ðŸŽ“ Tag Usage Guidelines

> [!core-principle]
> **Consistency Protocol for PKM Domain:**

### Rule 1: Meta-Documentation Standard
Every system component (workflow, template, configuration) should have documentation tagged appropriately:
- The component itself (e.g., a Templater script)
- Documentation about the component (tagged `#type/reference`)
- Reflections on using the component (tagged `#type/reflection`)

### Rule 2: Distinction Between Tool and Practice
- Use `#obsidian/` tags for tool-specific implementation
- Use `#pkm/` or `#knowledge-workflow/` for methodology that transcends any specific tool
- Example: "How to create atomic notes" = `#pkm/principles/atomic-notes`
- Example: "Using Obsidian's block references" = `#obsidian/core-features/linking`

### Rule 3: Methodology vs. Implementation
- Tag notes about *theories and frameworks* with methodology tags
- Tag notes about *actually doing it* with workflow tags
- Example: "PARA Method explained" = `#productivity/para` + `#type/framework`
- Example: "My PARA implementation" = `#productivity/para` + `#pkb/architecture` + `#type/practice-log`

### Rule 4: Avoid Tool Brand Proliferation
- Keep tool-specific tags under the `#obsidian/` hierarchy
- Don't create separate top-level tags for every plugin
- Bad: `#dataview`, `#templater`, `#quickadd`
- Good: `#obsidian/plugins/dataview`, `#obsidian/plugins/templater`

### Rule 5: Context Tags for Meta-Work
When working on the system itself (vs. using the system), use `#context/meta`:
```yaml
# Working on improving the tag system:
tags:
  - type/reflection
  - pkb/metadata/tags
  - context/meta

# Actually using tags in normal workflow:
tags:
  - type/literature
  - cognitive-science/metacognition
  - context/theoretical
```

### Rule 6: Status Matters for Documentation
System documentation should have clear status tags:
- `#status/evergreen` = Authoritative, current documentation
- `#status/review` = Potentially outdated, needs verification
- `#status/archived` = Historical record, no longer active

## âš ï¸ Common Tagging Mistakes in This Domain

> [!warning]
> **Anti-Patterns to Avoid in PKM Tagging:**

### âŒ Mistake 1: Confusing the Map with the Territory

**Bad:**
```yaml
# A note about cognitive psychology
tags:
  - pkm
  - note-taking
```

**Good:**
```yaml
# A note about cognitive psychology
tags:
  - cognitive-science/memory
  - type/literature
```

**Why:** PKM tags describe notes *about knowledge management*, not notes *managed by* your knowledge system. Every note in your PKB is managed by PKMâ€”that doesn't mean every note should be tagged `#pkm`.

---

### âŒ Mistake 2: Tool-Specific When Methodology Applies

**Bad:**
```yaml
# A note about linking practices in general
tags:
  - obsidian/core-features/linking
```

**Good:**
```yaml
# A note about linking practices in general
tags:
  - knowledge-graph/linking
  - pkm/principles/linking-thinking
  - concept/bidirectional-linking
```

**Why:** Linking is a methodology that transcends Obsidian. Only use `#obsidian/` tags for Obsidian-specific implementation details.

---

### âŒ Mistake 3: Documentation About Practice, Not the Practice Itself

**Bad:**
```yaml
# Documentation explaining your weekly review
tags:
  - type/practice-log
```

**Good:**
```yaml
# Documentation explaining your weekly review
tags:
  - type/framework
  - knowledge-workflow/review/weekly-review
  - status/evergreen
```

**Why:** `#type/practice-log` is for records of doing the practice, not for documentation of how to do it.

---

### âŒ Mistake 4: Over-Tagging System Components

**Bad:**
```yaml
# A Templater script file
tags:
  - obsidian/plugins/templater
  - obsidian/advanced/templater-scripts
  - automation
  - workflow
  - pkm
  - productivity
```

**Good:**
```yaml
# A Templater script file
tags:
  - type/template
  - obsidian/plugins/templater
```

**Why:** Templates and scripts are functional components. Keep tagging minimal and specific.

---

### âŒ Mistake 5: Meta Confusion

**Bad:**
```yaml
# A note about your tag taxonomy (like this one!)
tags:
  - type/reference
  - pkb/metadata/tags
```

**Good:**
```yaml
# A note about your tag taxonomy (like this one!)
tags:
  - type/reference
  - pkb/metadata/tags
  - context/meta
  - status/evergreen
```

**Why:** Always add `#context/meta` when a note is about the system itself. This distinguishes meta-work from regular work.

---

### âŒ Mistake 6: Plugin Proliferation

**Bad:**
Creating separate top-level tags:
```yaml
tags:
  - dataview
  - templater
  - quickadd
  - calendar
```

**Good:**
Using the plugin hierarchy:
```yaml
tags:
  - obsidian/plugins/dataview
  - obsidian/plugins/templater
```

**Why:** Keeps your tag namespace clean and makes it clear these are tool-specific, not conceptual categories.
